============================================================
教材化補強章節：如何閱讀 OpenCLI 的控制迴圈
============================================================

這一章不只是補充技術細節，而是把原本偏工程師視角的說明，改寫成更像教學講義的形式。目標是讓讀者不需要先熟悉 OpenCLI 原始碼，也能理解：

1. 這段程式碼在整體系統裡的位置
2. 它為什麼會形成「迴圈」
3. 這些迴圈如何真的影響 `chatgpt-web ask`
4. 遇到問題時應該先看哪一層

------------------------------------------------------------
一、先不要急著看程式碼
------------------------------------------------------------

很多人看到 `goto()`、`evaluate()`、`sendCommand()` 這些函式時，第一反應會是：「這不就是一些工具函式嗎？」

但在 OpenCLI 裡，它們不是普通 helper，而是整個 browser automation 的控制節點。

你可以把 OpenCLI 想成三層：

第一層：你下命令
- 例如：`opencli chatgpt-web ask "今天天氣如何？"`

第二層：Page 抽象層負責翻譯命令
- 例如：`goto()`、`evaluate()`、`insertText()`

第三層：daemon 與 Chrome 真正執行
- daemon 收到命令
- extension / CDP 接手
- Chrome 頁面實際被控制

所以，**Page 不是瀏覽器本身，而是 CLI 與瀏覽器之間的翻譯層。**

------------------------------------------------------------
二、先用一張腦中地圖理解
------------------------------------------------------------

請把整體流程先想成下面這張腦中圖：

CLI 命令
  ↓
Adapter
  ↓
Page abstraction
  ↓
sendCommand(...)
  ↓
Daemon
  ↓
Extension / CDP
  ↓
Chrome
  ↓
頁面結果回傳

這張圖最重要的意義是：

- Adapter 不直接碰 Chrome
- Page 也不直接碰 Chrome
- 真正執行的是 daemon + extension / CDP
- 但 adapter 與 Page 決定了要執行什麼、怎麼輪詢、何時停止

------------------------------------------------------------
三、教材式閱讀法：先看角色，再看程式
------------------------------------------------------------

如果直接把原始碼貼上來，很多讀者會立刻迷失在細節裡。

比較好的閱讀方式是：

### 先問：這段程式碼的角色是什麼？

以 `goto(url)` 為例，它的角色不是「打開網頁」這麼簡單，而是：

- 把導頁需求送給 daemon
- 取得目前正確的 tabId
- 確保後續命令都打到同一個分頁
- 在必要時等待頁面穩定

所以你在讀 `goto()` 時，不要只看「它調了哪些 API」，要看它在**控制鏈**裡扮演什麼角色。

------------------------------------------------------------
四、範例一：如何讀 `goto()`
------------------------------------------------------------

原始概念：

```js
async goto(url, options) {
    const result = await sendCommand('navigate', {
        url,
        ...this._cmdOpts(),
    });

    if (result?.tabId) {
        this._tabId = result.tabId;
    }

    this._lastUrl = url;

    if (options?.waitUntil !== 'none') {
        await sendCommand('exec', { code: combinedCode, ...this._cmdOpts() });
    }
}
```

### 先不要急著逐行翻譯
先看它在做哪四件事：

1. 把導頁命令送出去
2. 記住現在真正操作的是哪個 tab
3. 記住最後一次打開的 URL
4. 視需要檢查頁面是否已穩定

### 再逐段看

`sendCommand('navigate', ...)`
- 白話：請 daemon 幫我打開這個網址

`if (result?.tabId) { this._tabId = result.tabId; }`
- 白話：Chrome 告訴我剛剛其實打開的是哪個分頁，我要把它記下來
- 教學重點：如果不記這個 tabId，後面很容易命令送錯頁

`this._lastUrl = url;`
- 白話：記住最後開過哪個網址
- 教學重點：這不是最核心，但對 debug 很有幫助

`sendCommand('exec', ...)`
- 白話：再送一段程式到頁面裡，確認頁面狀態穩定
- 教學重點：OpenCLI 不把「頁面打開」等同於「頁面可操作」

### 這一段的教材重點
請記住一句話：

**`goto()` 不只是導頁，它其實在建立「後續所有操作應該落在哪個 tab、在哪個狀態下繼續執行」的基礎。**

------------------------------------------------------------
五、範例二：如何讀 `evaluate()`
------------------------------------------------------------

原始概念：

```js
async evaluate(js) {
    const code = wrapForEval(js);
    try {
        return await sendCommand('exec', { code, ...this._cmdOpts() });
    } catch (err) {
        if (!isRetryableSettleError(err)) throw err;
        await new Promise((resolve) => setTimeout(resolve, 200));
        return sendCommand('exec', { code, ...this._cmdOpts() });
    }
}
```

### 這段在做什麼？

不要把它只看成「執行 JavaScript」。
它真正的角色是：

- 把頁面端程式碼包裝好
- 透過 daemon 丟進正確 tab
- 如果頁面 target 暫時失效，就等一下再重試

### 教材式拆法

`wrapForEval(js)`
- 白話：把你要在頁面裡跑的程式碼包裝成可執行形式

`sendCommand('exec', ...)`
- 白話：請 daemon 到 Chrome 裡幫我執行這段 code

`catch (err)` + retry
- 白話：如果頁面正在切換或 SPA target 暫時不穩，不要立刻放棄，再試一次

### 教學重點
這說明了 OpenCLI 的設計假設：

**現代網頁不穩定是常態，不是例外。**

所以 `evaluate()` 的核心不是「幫你跑 JS」，而是「幫你在不穩定頁面裡，盡量可靠地跑 JS」。

------------------------------------------------------------
六、兩種迴圈一定要分清楚
------------------------------------------------------------

讀這份文件時，最容易混淆的就是「到底哪裡叫迴圈？」

這裡其實有兩種不同層級：

### 迴圈 A：命令轉發迴圈

這是這樣的：

CLI → Page → daemon → extension / CDP → Chrome → 回傳結果

每次 `goto()`、`evaluate()`、`screenshot()` 都會走一輪。

這不是 `for` 或 `while` 的程式迴圈，而是控制流程上的閉環。

### 迴圈 B：狀態輪詢迴圈

這才是你在 code 裡會明顯看到的：

- 讀狀態
- 檢查條件
- 等一下
- 再讀一次
- 直到成功或 timeout

例如：
- `waitForReady()`
- `waitForAssistantResponse()`

### 最重要的教學句

**命令轉發迴圈決定命令送不送得進去，狀態輪詢迴圈決定結果抓不抓得回來。**

------------------------------------------------------------
七、為什麼 `chatgpt-web ask` 特別依賴這兩種迴圈
------------------------------------------------------------

ChatGPT Web 不是靜態頁面，它有很多動態行為：

- React state 改變
- textarea 值更新
- send button enable/disable
- assistant 回應逐步生成
- 對話內容持續變化

這意味著：

只送一次命令，不夠。
只讀一次 DOM，也不夠。

所以 `ask` 才會長成這種形式：

1. `goto(ChatGPT)`
2. `waitForReady()`
3. `clickNewChat()`
4. `focusComposerAndType()`
5. `submitComposer()`
6. `waitForAssistantResponse()`
7. 回傳結果

### 教學重點
這不是「流程很多」，而是因為每一段都在處理不同的不確定性：

- 頁面是否已準備好
- 輸入框是否真的存在
- prompt 是否真的進入 React state
- submit 是否真的觸發
- assistant 是否真的開始生成
- 什麼時候算生成完成

------------------------------------------------------------
八、常見誤解
------------------------------------------------------------

### 誤解 1：只要 selector 對了就會成功
不對。selector 對了，只代表你找到了元素，不代表前端 state 真的更新。

### 誤解 2：DOM value 改了就算輸入成功
不對。React / SPA 網頁常常需要更完整的事件鏈。

### 誤解 3：只要讀最後一段文字就是答案
不對。最後一段可能是舊內容、空白 placeholder，或尚未穩定的半句。

### 誤解 4：Page 是瀏覽器本身
不對。Page 是控制抽象層，真正執行命令的是 daemon 與 Chrome。

------------------------------------------------------------
九、如果讀者只記一件事
------------------------------------------------------------

如果整段看完只記一句話，最應該記這句：

**OpenCLI 的關鍵，不在於它有多少 helper function，而在於它把「命令送進瀏覽器」與「等待頁面進入正確狀態」這兩件事，做成可反覆執行的控制迴圈。**

這也是為什麼 `chatgpt-web ask` 能成立的真正原因。
